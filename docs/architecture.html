Shared Memory Server

Architecture for memory.example.com

Express + Node.js SQLite + FTS5 MCP Protocol Bearer Auth File Sync

Architecture Diagram

Overview

A shared remote memory server enabling Claude Code and Claude Desktop to share persistent memory. Claude Code sessions generate observations, session summaries, and user prompts locally. A sync daemon pushes these to the remote server, along with auto-synced project .md files. Claude Desktop connects via a local MCP stdio proxy that bridges to the remote MCP endpoint, creating a bidirectional knowledge bridge.

Deployed on Your Server at memory.example.com, port 3010, inside a server process behind reverse proxy reverse proxy with automatic HTTPS.

Data Flows

  • Code → Remote (DB Sync): The sync daemon (push-to-remote.js) polls the local SQLite database every 30 seconds and POSTs new observations, session summaries, and user prompts to the remote API. Batches of up to 100 records. Handles 409 duplicates gracefully.
  • Code → Remote (File Sync): Every 5 minutes, the sync daemon scans the project for .md files and syncs them to the remote as type="reference" observations with source="file-sync". Files >100KB are skipped. Uses upsert to update existing records when file contents change.
  • Code ← Remote (Pull): Deferred — planned for a session-start hook that supplements Claude Code's local context with remote memories from other sources.
  • Desktop ↔ Remote (MCP via Local Proxy): Claude Desktop connects through a local stdio proxy (mcp-local-proxy.js) that forwards JSON-RPC 2.0 messages to the remote POST /mcp endpoint over HTTPS. Four tools available: search, timeline, get_observations, save_memory.

File Sync

Auto-sync project .md files

The sync daemon automatically indexes all .md files from the project directory, making them searchable via MCP from Claude Desktop.

  • Runs every 5 minutes (separate from the 30-second DB sync cycle)
  • Files are stored as observations with type="reference" and source="file-sync"
  • Files larger than 100KB are skipped
  • Uses upsert semantics: same file path = same source_id, so content updates replace the previous version
  • The FTS5 index is rebuilt on upsert, keeping search results current

Local MCP Proxy

Stdio bridge for Claude Desktop

Claude Desktop requires MCP servers to be local stdio processes — it cannot connect to remote HTTP URLs directly. The local MCP proxy (sync/mcp-local-proxy.js) bridges this gap.

  • Reads JSON-RPC 2.0 messages from stdin
  • Forwards each request to the remote server over HTTPS (POST /mcp)
  • Writes responses back to stdout
  • Auth credentials loaded from ~/.claude-mem/remote-sync-env
  • Configured in Claude Desktop's MCP settings as a stdio command

Upsert Mechanism

Update-on-duplicate for observations

POST /api/observations?upsert=true enables update-on-duplicate behavior.

  • When a record with the same (source, source_id) already exists, the existing record is updated instead of returning 409 Conflict
  • The FTS5 full-text search index is rebuilt for the updated record
  • Without ?upsert=true, the default 409 behavior is preserved
  • Primarily used by file sync to update file contents when they change on disk

Sync Components

push-to-remote.js

Main sync daemon. Performs two jobs:

  • DB sync — every 30 seconds, pushes new observations, sessions, and prompts
  • File sync — every 5 minutes, syncs all .md files as reference observations

mcp-local-proxy.js

Stdio proxy for Claude Desktop. Reads JSON-RPC from stdin, forwards to remote HTTPS endpoint, writes responses to stdout.

  • Auth from ~/.claude-mem/remote-sync-env

com.claude-mem-sync.plist

macOS LaunchAgent that keeps the sync daemon running persistently.

  • Auto-starts on login
  • Restarts on crash

MCP Tools

search

Full-text search across observations and session summaries using FTS5.

  • query (required) — search string
  • limit — max results (default 20, max 100)
  • project — filter by project

timeline

Chronological timeline with anchor-based navigation.

  • anchor — obs ID, S{id}, or ISO timestamp
  • query — FTS query to find anchor
  • depth_before / depth_after
  • project — filter by project

get_observations

Batch fetch observations by their IDs.

  • ids (required) — array of observation IDs

save_memory

Save a new memory from Claude Desktop. Stored as observation with source claude-desktop.

  • text (required) — stored as narrative
  • title — defaults to first 80 chars
  • type — defaults to "discovery"
  • project — defaults to "desktop"

API Reference

Public Endpoints

MethodPathDescription
GET /api/health Health check. Returns status, version, observation count, disk free MB.

Authenticated Endpoints

All require Authorization: Bearer <API_KEY>. Rate limited to 600 requests per 60-second window (increased from 120 to handle bulk file sync).

MethodPathDescription
POST /api/observations Create an observation. Requires source. Returns { id, success }. 409 on duplicate. Supports ?upsert=true query param to update existing record on duplicate instead of 409; FTS5 index is rebuilt on upsert.
POST /api/sessions Create a session summary. Requires source. Returns { id, success }. 409 on duplicate.
POST /api/prompts Create a user prompt. Requires source. Returns { id, success }. 409 on duplicate.
POST /api/observations/batch Batch fetch observations by ID array. Body: { ids: [1, 2, 3] }.
GET /api/search Full-text search. Params: query (required), project, type, obs_type, limit, offset.
GET /api/timeline Chronological timeline. Params: anchor, query, depth_before, depth_after, project.
GET /api/context Markdown context for specified projects. Params: projects (comma-separated, required).
POST /mcp MCP JSON-RPC 2.0 endpoint. Methods: initialize, ping, tools/list, tools/call.

Sync Process

The sync daemon (sync/push-to-remote.js) runs as a macOS launchd service (com.claude-mem-sync.plist).

How it works

  1. Opens local ~/.claude-mem/claude-mem.db in read-only mode
  2. Loads sync cursor from remote-sync-state.json (last synced ID per table)
  3. Queries new records: WHERE id > lastSyncedId LIMIT 100
  4. Resolves cross-table references (observation → session via memory_session_id; prompt → session via sdk_sessions)
  5. POSTs each record to the remote API
  6. 409 Conflict treated as success (record already exists)
  7. Saves updated cursor, sleeps 30s, repeats
  8. Every 5 minutes, scans for .md files and syncs them with ?upsert=true (file sync cycle)
